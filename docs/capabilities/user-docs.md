# User Docs

The `/opsx:docs` command generates user-facing documentation for each capability in the project, producing one file per capability at `docs/capabilities/<capability>.md` with features, behavior, edge cases, and -- when change data is available -- context about purpose, design rationale, limitations, and future enhancements.

## Purpose

Without generated capability documentation, understanding what each capability does, how it behaves, and what its limitations are requires reading formal specification files full of normative language and Gherkin scenarios. User docs translate specifications into clear, natural explanations so that anyone can understand capabilities without navigating spec files or digging through change artifacts for design context.

## Rationale

Documentation is generated from baseline specs as the source of truth, with enrichment from completed change artifacts (proposal, research, design, preflight) that provide context about why capabilities exist and what trade-offs were made. Purpose sections use problem-framing derived from the spec's Purpose section rather than proposal motivations. Rationale sections describe the current design in present tense to explain why things work the way they do, avoiding change-history narration. A section-completeness rule ensures all template sections are included when source data exists. Incremental generation uses the spec's `lastModified` frontmatter field for direct comparison against each doc's `lastUpdated` -- no change directory scanning or proposal parsing is needed for change detection. Content-aware writes prevent false timestamp bumps when regeneration produces identical output.

## Features

- **One doc per capability** -- each baseline spec produces a documentation file at `docs/capabilities/<capability>.md`
- **User-facing language** -- normative specification language (SHALL/MUST) is replaced with natural explanations
- **Scenario-to-behavior conversion** -- Gherkin scenarios become readable usage descriptions
- **Change enrichment** -- when completed changes exist for a capability, the doc gains Purpose, Rationale, Known Limitations, and Future Enhancements sections
- **Problem-framed Purpose** -- derived from the spec's Purpose section, framing what goes wrong without the capability
- **Present-tense Rationale** -- explains why the design works this way, not how it evolved
- **Known Limitations** -- derived from design Non-Goals (technical constraints), Risks, and preflight assumptions
- **Future Enhancements** -- derived from deferred Non-Goals and items marked for separate implementation
- **Spec-only fallback** -- capabilities without change data still get documentation from the spec alone, including Rationale when design reasoning is derivable
- **Multi-command workflow notes** -- capabilities spanning multiple commands include a workflow sequence note and command names in behavior headers
- **Spec-based incremental detection** -- compares each spec's `lastModified` frontmatter against the doc's `lastUpdated` to decide whether regeneration is needed, without scanning change directories
- **Multi-capability mode** -- pass a comma-separated list of capability names to regenerate only specific capabilities, skipping the full incremental scan
- **Content-aware writes** -- generated content is compared against existing files to prevent false timestamp updates
- **Step independence** -- each documentation step reads its own sources, producing consistent results in any execution context
- **Language support** -- all content can be generated in the language configured via `docs_language`

## Behavior

### Documentation from Spec

For each baseline spec, the agent creates a documentation file containing a description of the capability, explanations of each feature, and usage descriptions derived from the scenarios.

### User-Facing Language

Specification language like "The system SHALL validate email format before account creation" becomes natural language such as "Email addresses are validated when you create an account."

### Gherkin Scenarios as Usage Descriptions

Scenarios described with GIVEN/WHEN/THEN in the spec are converted into natural descriptions of the workflow or behavior.

### Implementation Details Excluded

Internal module names, database schema details, and other implementation specifics are omitted. Documentation focuses only on user-visible behavior.

### Change-Enriched Documentation

When completed changes exist for a capability, the agent reads proposal.md, research.md, design.md, and preflight.md to enrich the doc with Purpose, Rationale, Known Limitations, and Future Enhancements sections. Change proposals are identified via their frontmatter `capabilities` field (falling back to parsing the Capabilities section for legacy proposals without frontmatter).

### Spec-Only Fallback

Capabilities with no completed changes get documentation with Overview, Features, Behavior, and Edge Cases sections. Rationale is included when design reasoning can be derived from spec requirements and scenarios.

### Purpose Uses Problem-Framing

The Purpose section is derived from the spec's Purpose section using problem-framing -- describing what goes wrong without this capability. It is not derived from proposal "Why" sections and does not restate the spec Purpose verbatim.

### Initial-Spec-Only Rationale from Spec

For capabilities whose only change is the initial bootstrap, Rationale is derived from spec requirements and scenarios, explaining why the design works this way.

### Rationale in Present Tense

Rationale describes the current design in present tense without narrating change history.

### Multi-Command Workflow Note

Capabilities that span multiple commands include a brief workflow sequence note at the top of the Behavior section explaining when each command is used.

### Multi-Command Behavior Headers

For multi-command capabilities, behavior subsection headers include the command name for quick scanning.

### Edge Cases Contain Only Surprising Behavior

The Edge Cases section includes only surprising states, error conditions, or non-obvious interactions. Normal flow variants are placed in the Behavior section.

### Conciseness Guards

Purpose contains at most 3 sentences, Rationale at most 3-5 sentences, and Known Limitations at most 5 bullets.

### Section Completeness

All sections from the template are included when source data exists. A section is only omitted when no source data is available.

### Behavior Subsections Match Scenario Groups

Each distinct Gherkin scenario group in the spec produces at least one Behavior subsection.

### Template-Driven Format

The agent reads the capability doc template at `openspec/templates/docs/capability.md` and uses it as the structural format for the output.

### Skipping Unchanged Capabilities

Before generating each capability doc, the agent reads the spec's `lastModified` frontmatter and compares it against the doc's `lastUpdated` frontmatter. If the spec's `lastModified` is older than the doc's `lastUpdated`, the capability is skipped entirely -- no change reading, no generation, no file write. If the spec has no `lastModified` field, it is treated as needing regeneration.

### Regenerating When Spec Is Newer

When a spec's `lastModified` is newer than or equal to the doc's `lastUpdated`, the capability is regenerated.

### First Run Generates All

When no capability docs exist yet, all capabilities are generated (equivalent to a full generation run).

### Content-Aware Write Prevents False Timestamp Bump

After generating a capability doc, the agent compares the generated content (excluding the `lastUpdated` field) against the existing file. If the content is identical, the file is not written and the timestamp is not updated.

### Single-Capability Mode Scopes Change Reading

When you provide a capability name argument (for example, `/opsx:docs release-workflow`), only completed changes matching that capability are read. The specified capability is always regenerated regardless of dates.

### Multi-Capability Mode Skips Full Scan

When you provide a comma-separated list of capability names, only the listed capabilities are processed. Each is always regenerated regardless of change dates. If a listed capability does not exist, you are warned and it is skipped. This mode is designed for the post-apply workflow where affected capabilities are already known.

### Output Summary

After generation, the agent reports which capabilities were regenerated, which were skipped due to unchanged content, and which were skipped because the spec was not modified.

### Documentation in Configured Language

When `docs_language` is set, headings and content are generated in the target language. YAML frontmatter keys remain in English while `title` and `description` values are translated. Product names, commands, and file paths remain in English.

### Default to English

When the `docs_language` field is missing, all documentation is generated in English.

### Product Names Preserved

Product names (OpenSpec, Claude Code), commands (`/opsx:docs`), and file paths remain in English regardless of the configured language.

### Language Change Triggers Full Regeneration

If existing docs were generated in one language and `docs_language` is changed to another, all docs are fully regenerated in the new language rather than incrementally updated.

## Known Limitations

- Does not detect spec-only changes without a `lastModified` update -- if a spec is modified without updating `lastModified` (for example, a manual edit outside the workflow), the capability doc is not regenerated until the field is updated. You can force regeneration with `/opsx:docs <capability>`.
- LLM translation quality varies by language; major languages produce high-quality results, while less common languages may need manual review.
- Content depth (such as Behavior detail and Edge Cases completeness) may be thinner when docs are generated from scratch without an existing doc as a quality baseline.

## Future Enhancements

- Full per-step restructure for autonomous agent readiness, where each documentation step is a fully self-contained unit with its own read instructions.
- Preflight validation for decision granularity to catch excessively fine-grained decisions before they reach the docs pipeline.

## Edge Cases

- If `openspec/specs/` is empty, the agent informs you and suggests running `/opsx:setup` first.
- If a spec has no scenarios, documentation is still generated from requirement descriptions, noting that usage examples are unavailable.
- If a change lacks design.md, research.md, or preflight.md, the agent skips enrichment from that artifact without error.
- If every Non-Goal in the design is a change-level scope boundary, Known Limitations is omitted.
- If an unsupported or misspelled language is configured, the agent makes a best-effort attempt and warns if the language is unrecognizable, falling back to English.
- If a capability doc exists but has no `lastUpdated` field or a malformed value, it is treated as needing regeneration.
- If the spec has no `lastModified` field, it is treated as needing regeneration (unknown modification date).
- If a comma-separated capability list contains duplicates, each capability is processed only once.
- If whitespace surrounds capability names in a comma-separated list, it is trimmed automatically.
