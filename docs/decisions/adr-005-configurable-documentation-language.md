# ADR-005: Configurable Documentation Language

## Status

Accepted — 2026-03-05

## Context

All user-facing documentation generated by `/opsx:docs` and `/opsx:changelog` was produced exclusively in English, preventing non-English-speaking teams from using the documentation output in their preferred language. The plugin's configuration surface (`openspec/config.yaml`) already supported a `context` field passed to all skills but had no language setting. Three approaches were investigated: a central `docs_language` field in config.yaml, a language setting in the constitution, and per-skill language parameters. The constitution approach was rejected because it mixes configuration with project-specific governance rules, and per-skill parameters would introduce duplication requiring users to set the same value in multiple places. A key constraint was that internal workflow artifacts (research, proposal, specs, design, preflight, tasks) must remain in English since they feed into the artifact pipeline and are consumed by other skills. Doc templates at `openspec/schemas/opsx-enhanced/templates/docs/` are structural guides and must also remain in English regardless of output language.

## Decision

1. **Single `docs_language` field in config.yaml** — Central, backward-compatible configuration read by all skills via existing config loading. Missing field defaults to English, ensuring full backward compatibility with existing projects.

2. **Commented-out field in init template for discoverability** — Users discover the feature without it being active by default. Existing projects are unaffected, avoiding unnecessary noise in configuration files.

3. **English enforcement via config `context` field** — The context field is passed to all skills automatically, providing a single enforcement point without per-skill duplication. This keeps internal workflow artifacts in English regardless of the docs_language setting.

4. **Translation at generation time, not in templates** — Templates are structural guides. Translating during generation allows one set of templates for all languages, avoiding template proliferation and maintenance burden across multiple language variants.

## Alternatives Considered

- Per-skill language parameter — rejected because it duplicates configuration across multiple skills and requires users to set the same value in several places
- Environment variable for language — rejected because it is not persistent across sessions
- Constitution entry for language setting — rejected because it mixes configuration concerns with governance rules
- Active field defaulting to English — rejected because it adds unnecessary noise to config for users who do not need the feature
- Separate config documentation for discoverability — rejected because a commented-out field in the template is more discoverable
- Per-skill English enforcement instruction — rejected because it requires updating every skill individually
- Schema-level language rule — rejected because language is a generation-time concern, not a pipeline concern
- Per-language template sets — rejected because of template proliferation and maintenance burden

## Consequences

### Positive

- Non-English-speaking teams can generate documentation in their preferred language with a single configuration change
- Full backward compatibility — existing projects without the field continue to generate English documentation
- Single enforcement point for English workflow artifacts via the config context field
- One set of templates serves all languages, minimizing maintenance burden
- YAML frontmatter keys remain English (machine-readable) while user-facing values get translated

### Negative

- LLM translation quality varies by language — major languages (German, French, Spanish) produce high-quality results, but exotic languages may need manual review
- No runtime validation of the `docs_language` value — mitigated by LLMs handling fuzzy language names well, with worst case falling back to English
- Language change mid-project leaves mixed-language changelog entries — existing entries are preserved in the previous language while new entries use the new language
