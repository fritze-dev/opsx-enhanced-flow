# ADR-034: Single docs_language field in config.yaml

## Status
Accepted (2026-03-05)

## Context
All user-facing documentation generated by `/opsx:docs` and `/opsx:changelog` was English-only, preventing non-English-speaking teams from using the documentation in their preferred language. The plugin needed a language configuration mechanism that is central, backward-compatible, and accessible to all documentation-generating skills. Research evaluated three approaches: a `docs_language` field in config.yaml (central, already read by all skills), a constitution entry (mixes configuration with governance rules), and per-skill parameters (duplication). The config.yaml approach leverages the existing configuration loading mechanism — all skills already read config.yaml at startup, so no new integration is needed. Missing the field defaults to English, ensuring full backward compatibility.

## Decision
Add a single `docs_language` field in config.yaml as the central language configuration for all documentation output.

## Rationale
Central, backward-compatible (missing defaults to English), and all skills already read config. Single configuration point avoids duplication.

## Alternatives Considered
- Per-skill parameter — creates duplication across multiple skills
- Environment variable — not persistent across sessions
- Constitution entry — mixes configuration concerns with governance rules

## Consequences

### Positive
- Single configuration point for all documentation language
- Fully backward-compatible: existing projects are unaffected
- No new integration needed since skills already read config.yaml

### Negative
- LLM translation quality varies by language; acceptable for major languages, exotic languages may need manual review
- No runtime validation of the field value; mitigated by LLMs handling fuzzy language names well
