# ADR-013: Fix ADR Reference Quality

## Status

Accepted — 2026-03-23

## Context

An audit of 12 consolidated ADRs generated by `/opsx:docs` revealed three categories of reference quality issues: broken spec links pointing to renamed specs (1 ADR referenced `docs-generation` which had been split into `user-docs`, `architecture-docs`, and `decision-docs`), external GitHub URLs with incorrect org names (3 ADRs used `robinfritze` instead of `fritze-dev`), and missing cross-references between thematically related ADRs (6 ADRs lacked back-references to the systems they modified). The root cause for the URL problem was that external links in ADR References are inherently fragile — archive content may contain stale URLs, and there is no single source of truth for the correct format. The archive backlink already provides a traceability chain to GitHub issues (ADR → Archive → proposal.md → issue reference), making direct issue links in ADRs redundant. For spec links and archive links, post-generation validation via glob can catch broken references before they are committed.

## Decision

1. **Internal-only references in ADRs** — ADR References contain only internal relative links (archive backlinks, spec links, ADR cross-references). No external URLs (GitHub issues, external docs). Traceability to issues is preserved via the archive backlink chain. This eliminates external URL maintenance entirely.

2. **Post-generation reference validation via glob** — After writing each ADR's References, every `[Spec: <name>]` link is validated by globbing `openspec/specs/<name>/spec.md`, and every `[Change: <name>]` link is validated against existing change directories. Broken links are replaced with successors or flagged with `<!-- REVIEW -->` markers. This catches renames and splits that would otherwise produce dead links.

3. **Heuristic cross-referencing for related ADRs** — Beyond same-archive cross-references, the agent checks if the current archive references other changes by name or modifies the same capabilities as earlier archives. When a clear thematic relationship is evident, a back-reference is added. Cross-references are not added speculatively. This catches obvious relationships without over-linking.

## Alternatives Considered

- **GitHub URL normalization from plugin.json** — would still require maintaining external URLs; doesn't eliminate the root cause of stale URLs
- **Keep external URLs with validation** — adds complexity without removing the maintenance burden; URLs can still drift between runs
- **Manual review only for broken links** — error-prone and not scalable as archive count grows
- **Full graph analysis for cross-references** — too complex for prompt-based instructions; heuristic approach is simpler and sufficient

## Consequences

### Positive

- External URL maintenance eliminated entirely — no more wrong org names or stale GitHub links
- Broken spec links caught automatically during generation — renamed or split specs produce valid references
- Change links verified to exist — prevents references to non-existent change directories
- Related ADRs are connected via cross-references — readers can follow the evolution of a system across multiple decisions

### Negative

- Less direct traceability to GitHub issues — readers must follow the archive backlink and then read proposal.md to find issue references (one extra hop)
- Cross-reference heuristic may miss some relationships when connections are not explicit in archive content — manual review can supplement
- Existing ADRs that previously contained GitHub issue links lose those links on regeneration
