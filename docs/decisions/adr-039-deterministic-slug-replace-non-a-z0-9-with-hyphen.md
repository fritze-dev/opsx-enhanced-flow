# ADR-039: Deterministic slug: replace non-[a-z0-9] with hyphen

## Status
Accepted (2026-03-05)

## Context
ADR file slugs were generated by an underspecified rule ("lowercase, replace spaces with hyphens") that produced different results across runs for decision titles containing dots, parentheses, slashes, or other special characters. For example, `/opsx:sync` could become `opsxsync` in one run and `opsx-sync` in another. This non-determinism caused file renames on each regeneration, creating unnecessary git churn. Research confirmed that a deterministic algorithm handling all special characters uniformly would produce consistent results: lowercase the string, replace any character not in `[a-z0-9]` with a hyphen, collapse consecutive hyphens, trim leading/trailing hyphens, truncate to 50 characters, and trim trailing hyphens again.

## Decision
Use a deterministic slug algorithm that replaces any character not in `[a-z0-9]` with a hyphen, collapses consecutive hyphens, trims, and truncates to 50 characters.

## Rationale
Handles all special characters uniformly. No ambiguity about dots, parentheses, or slashes. Produces consistent results across runs.

## Alternatives Considered
- Keep current underspecified rule — produces non-deterministic slugs across runs
- Add special-case rules per character type — complex and still potentially ambiguous

## Consequences

### Positive
- Same input always produces the same slug, eliminating unnecessary git churn
- Handles all special characters uniformly without edge cases

### Negative
- Slug change causes ADR file renames for existing ADRs (e.g., `opsxsync` becomes `opsx-sync`); all README links regenerate to match, so no broken links, but git shows renames
