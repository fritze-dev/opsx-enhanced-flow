# ADR-041: Skill Consolidation

## Status

Accepted (2026-04-09)

## Context

The opsx-enhanced plugin has 11 SKILL.md files totaling 1,593 lines, with 60-70% of their content duplicating what specs already describe. Shared logic like change context detection is copy-pasted across 6 skills. The three-layer architecture's third layer (Skills) consists of 13 commands spread across 11 files, creating a large surface area for maintenance and consistency issues. PR #97 validated the concept of sub-agent dispatch with action templates but created separate action template files, adding a fourth layer of files to maintain. The specs already serve as the single source of truth for all behavior -- skills are redundant procedural wrappers that restate spec content as instruction text. The investigation revealed that consolidating to a single router with 4 commands and inline action definitions in WORKFLOW.md would eliminate the duplication while making specs the true behavioral authority.

## Decision

1. **Replace 11 SKILL.md files with 1 router SKILL.md and 4 stub files** -- The plugin system requires one SKILL.md per command, so 4 stub files (each ~5 lines of frontmatter + include directive) point to a shared `router.md` (~80 lines). This reduces orchestration code from 1,593 lines to ~100 lines total, a 93% reduction. The router handles shared logic (intent recognition, change context detection, WORKFLOW.md loading) once.

2. **Reduce 13 commands to 4: init, propose, apply, finalize** -- `init` replaces setup + bootstrap. `propose` replaces new + ff + continue + discover + preflight (propose owns the full pipeline). `apply` replaces apply + verify (apply generates review.md as QA output). `finalize` replaces changelog + docs (finalize covers all post-approval work). The 4 commands map cleanly to lifecycle phases.

3. **Define actions inline in WORKFLOW.md, not as separate template files** -- Actions have no output structure (no body content to template), so separate files add maintenance without benefit. WORKFLOW.md body sections (`## Action: <name>`) contain `### Instruction` for procedural guidance. Requirement links live in the SKILL.md stubs, not in WORKFLOW.md, keeping prose out of frontmatter.

4. **Add review.md as the 7th pipeline artifact** -- review.md replaces the transient verify report with a persistent, PR-visible artifact. It is generated during apply as QA-loop output (not during artifact-forward generation). File existence serves as the non-skippable quality gate. research.md opens the change, review.md closes it.

5. **Eliminate discover entirely** -- 0 out of 49 changes used the discover command. Research is the first pipeline artifact generated by propose. Interactive Q&A happens naturally in conversation without a dedicated command.

## Alternatives Considered

- Single SKILL.md with all commands (plugin system may not support multiple commands per SKILL.md) -- rejected due to plugin system constraint
- All logic duplicated in stubs (no shared router.md) -- rejected because it recreates the duplication problem
- Separate action template files (PR #97 approach) -- rejected because actions have no body/output structure, making separate files unnecessary overhead
- Action definitions in specs -- rejected because actions are orchestration concern, not behavioral specification
- Transient verify report (current approach) -- rejected because transient output is lost after conversation, not PR-visible, and skippable
- `verify.md` naming instead of `review.md` -- rejected as less intuitive in PR review context
- Keep discover as `--discover` flag -- rejected because 0/49 usage shows the feature was never adopted
- template-version 2 (incremental) -- rejected because PR #97 already claimed v2 and this is a major architectural change

## Consequences

### Positive

- Orchestration code reduced by 93% (1,593 lines to ~100 lines), dramatically lowering maintenance burden
- Change context detection implemented once in the router instead of being copy-pasted across 6 skills
- Specs become the true single source of truth -- the router reads specs via requirement links, eliminating restated content in skills
- Pipeline commands reduce from 13 to 4, simplifying the user-facing command surface
- review.md as a persistent artifact makes verification results PR-visible and auditable
- All workflow configuration lives in one file (WORKFLOW.md) -- pipeline, actions, automation, worktree config
- Spec merges reduce total specs from 18 to 13, consolidating related concerns (project-setup + project-bootstrap = project-init; user-docs + architecture-docs + decision-docs = documentation; artifact-generation absorbed into artifact-pipeline)

### Negative

- Breaking change: consumers on template-version 1 must run `/opsx:workflow init` to migrate to template-version 3. Mitigated by init handling the migration.
- Plugin stub indirection adds a level of file-reading between command invocation and router logic. Mitigated by stubs being trivially simple (~5 lines each).
- Thin action instructions in WORKFLOW.md may be insufficient for complex actions. Mitigated by testing apply (the most complex action) first and expanding instructions if needed.
- review.md becomes stale after manual code changes post-verification. Mitigated by apply deleting and regenerating review.md on each run.
- Loss of standalone preflight and verify commands -- users must go through propose or apply. Accepted because 4-command sufficiency covers all workflow needs.
