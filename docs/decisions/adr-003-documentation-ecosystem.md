# ADR-003: Documentation Ecosystem

## Status

Accepted — 2026-03-04

## Context

The `/opsx:docs` skill read only baseline specs at `openspec/specs/*/spec.md` to generate user-facing capability docs, leaving valuable context from archived artifacts (proposal, research, design, preflight) entirely unused. Users could not understand why capabilities existed, what alternatives were considered, or what limitations applied without manually digging through archive directories. The existing `docs-generation` capability spec was also too broad, mixing user docs, changelog generation, and what would now include architecture overview and ADR generation in a single bloated spec. The research phase investigated three approaches: enriching `/opsx:docs` with all documentation tiers in a single skill (chosen), creating separate skills per tier like `/opsx:adr` and `/opsx:research-log` (rejected — skill proliferation, violates user preference), and enriching only capability docs while deferring ADRs (rejected — loses valuable output and requires multiple changes). Five archived changes were available for enrichment, and the archive structure consistently provided proposal.md, research.md, design.md, and preflight.md artifacts that could be parsed reliably. A key edge case was identified: the initial-spec archive's proposal "Why" section describes bootstrapping rather than individual capabilities, requiring special handling where spec Purpose is used instead for capabilities whose only archive is initial-spec.

## Decision

1. **Split `docs-generation` into `user-docs`, `architecture-docs`, and `decision-docs`** — Each concern is independently specifiable and testable. Changelog generation moves to `release-workflow` where it logically belongs. This replaces one bloated spec with three focused ones.

2. **All doc types generated by `/opsx:docs`, no new skills** — User preference, avoids skill proliferation, and provides a single entry point. Separate `/opsx:adr` and `/opsx:research-log` skills were rejected.

3. **Direct glob per capability instead of pre-built index** — The agent globs `openspec/changes/archive/*/specs/<capability>/` to find archives that touched each capability, then reads their artifacts. This is simpler than building a Capability-to-Change Index with proposal parsing, which was rejected as over-engineering given the small number of archives.

4. **ADRs fully regenerated each run** — Deterministic and reproducible with no state to track. Numbering is always consistent from chronological sort of archives. Incremental updates were rejected because they require tracking which archives have been processed.

5. **Research context integrated into ADR Context section** — Approaches evaluated and findings from research.md are woven into the ADR Context, providing a single place for "why did we decide this?" rather than maintaining a separate research log output.

6. **"Why This Exists" uses newest archive's proposal** — The most recent motivation is the most current; older motivations may have been superseded. Concatenating all proposals would be too verbose, and using only the oldest could be outdated.

7. **Initial-spec-only capabilities use spec Purpose for "Why This Exists"** — The initial-spec proposal describes bootstrapping the project, not individual capability motivations. Using it directly would be misleading, so spec Purpose provides the correct context for these capabilities.

## Alternatives Considered

- Keep everything in one `docs-generation` spec — too broad, mixes unrelated concerns, harder to test independently
- Separate `/opsx:adr` and `/opsx:research-log` skills — skill proliferation, user must remember multiple commands, violates user preference for single entry point
- Enrich only capability docs and defer ADR/research log — loses valuable output, requires multiple changes
- `--adr` and `--research` subcommand flags — unnecessary complexity given that full regeneration is fast
- Per-archive ADR numbering — non-standard; global sequential numbering is the convention
- Incremental ADR updates — requires state tracking for which archives are processed
- Capability-to-Change Index with proposal parsing — over-engineering for the number of archives
- Concatenate all proposals for "Why This Exists" — too verbose
- Use oldest proposal only — may be outdated
- Per-capability research logs — fragments the narrative that is naturally conducted at the change level
- Separate `docs/research/` output — more files, less focused than integrating into ADR Context
- Use initial-spec proposal "Why" for individual capabilities — misleading context about bootstrapping

## Consequences

### Positive

- Users can understand why capabilities exist, what alternatives were considered, and what limitations apply without digging through archives
- Architecture overview provides a cross-cutting view synthesized from constitution, three-layer-architecture spec, and all design decisions
- ADRs formalize the decision history in a standard format (Context, Decision, Consequences)
- Capability docs gain enriched sections (Why This Exists, Background, Known Limitations) derived from archive artifacts
- Single `/opsx:docs` command generates all documentation types in one run
- Focused specs (`user-docs`, `architecture-docs`, `decision-docs`) are each independently testable

### Negative

- SKILL.md prompt length roughly doubles (~150 to ~300 lines), though clear section headers maintain readability
- Missing archive artifacts (e.g., archives without design.md) require graceful fallback — the skill omits sections with no data rather than erroring
- ADR numbering could theoretically become unstable if archives were reordered, though in practice numbers are regenerated deterministically from chronological sort
