## Purpose

Manages the project constitution lifecycle including bootstrap-based generation from codebase observation, automatic updates during design phases, and global context enforcement across all AI actions.

## Requirements

### Requirement: Constitution Structure Defined by Schema Template
A recommended constitution section structure SHALL be provided as a template file at `openspec/templates/constitution.md`. The template SHALL contain section headings (`## Tech Stack`, `## Architecture Rules`, `## Code Style`, `## Constraints`, `## Conventions`, `## Standard Tasks`) with guidance comments describing what each section covers. The template SHALL define structure only — not content. The bootstrap skill SHALL read this template as a starting structure and adapt sections to fit the project — adding, omitting, or restructuring sections as appropriate.

**User Story:** As a schema maintainer I want the constitution section structure defined in a template file, so that it is consistent with the existing template system and can vary per schema.

#### Scenario: Bootstrap reads constitution template from schema
- **GIVEN** `openspec/templates/constitution.md` exists defining six sections
- **WHEN** the developer runs `workflow init`
- **THEN** the bootstrap skill reads the template as a starting structure
- **AND** the generated `openspec/CONSTITUTION.md` uses the template sections as a baseline, adapted to fit the project

#### Scenario: Template defines structure not content
- **GIVEN** the constitution template at `openspec/templates/constitution.md`
- **WHEN** its contents are inspected
- **THEN** it contains section headings and guidance comments describing what belongs in each section
- **AND** it does NOT contain project-specific content or sample values

#### Scenario: Sections adapted to project needs
- **GIVEN** a constitution template with recommended sections: Tech Stack, Architecture Rules, Code Style, Constraints, Conventions, Standard Tasks
- **WHEN** the bootstrap skill generates the constitution
- **THEN** the generated constitution may add, omit, or restructure sections based on the project's needs

### Requirement: Bootstrap-Generated Constitution
The project constitution (`openspec/CONSTITUTION.md`) SHALL be generated by the `workflow init` command based on observed patterns, conventions, and technologies found in the existing codebase. The bootstrap skill SHALL read the active schema's `templates/constitution.md` template as a starting structure and adapt sections to fit the project. The agent SHALL scan source files, configuration files, directory structures, and dependency manifests to infer content for each applicable section. The constitution SHALL NOT contain invented or aspirational rules -- every entry MUST be traceable to an observed pattern in the codebase. Items where the agent is uncertain about the convention or its scope SHALL be marked with a `<!-- REVIEW -->` comment. After generation, the agent SHALL iterate through all REVIEW markers, present each to the user for confirmation, document the user's decision, and remove the marker. No REVIEW markers SHALL remain in the final constitution after bootstrap completes.

**User Story:** As a developer bootstrapping an existing project I want the constitution to reflect what my codebase actually does, with its structure defined by the schema template and all uncertainties resolved through direct questions rather than invisible markers left for me to discover.

#### Scenario: Constitution generated from codebase scan
- **GIVEN** an existing project with TypeScript source files, a `tsconfig.json`, ESLint configuration, and a `package.json`
- **AND** the active schema has a `templates/constitution.md` defining the section structure
- **WHEN** the developer runs `workflow init`
- **THEN** the generated `openspec/CONSTITUTION.md` includes TypeScript under Tech Stack, references the ESLint rules under Code Style, and captures the project's module structure under Architecture Rules

#### Scenario: Uncertain conventions resolved through user interaction
- **GIVEN** a codebase where some files use tabs and others use spaces for indentation
- **WHEN** the agent generates the constitution
- **THEN** the agent initially marks the indentation entry with `<!-- REVIEW -->`
- **AND** the agent presents the inconsistency to the user: "Indentation convention unclear — both tabs and spaces observed. Which should be the standard?"
- **AND** after the user responds, the agent updates the entry with the user's decision and removes the `<!-- REVIEW -->` marker

#### Scenario: No invented rules
- **GIVEN** a Python project with no test framework configured
- **WHEN** the agent generates the constitution
- **THEN** the constitution does NOT include a testing convention or framework entry, because none was observed in the codebase

#### Scenario: Constitution covers applicable sections
- **GIVEN** a codebase with identifiable patterns
- **AND** the active schema's constitution template recommends sections: Tech Stack, Architecture Rules, Code Style, Constraints, Conventions, Standard Tasks
- **WHEN** the agent generates the constitution
- **THEN** the resulting constitution contains the applicable sections from the template, adapted to the project

#### Scenario: No REVIEW markers remain after bootstrap
- **GIVEN** a completed bootstrap run
- **WHEN** the final constitution is inspected
- **THEN** zero `<!-- REVIEW -->` markers remain in the file
- **AND** every uncertain item has been resolved with the user's explicit decision

### Requirement: Constitution as Global Context
The project constitution SHALL be referenced by all AI actions through the WORKFLOW.md `context` field. Every skill invocation and artifact generation step SHALL read the constitution before proceeding, ensuring that AI-generated content respects the project's established tech stack, architecture, code style, constraints, and conventions. The constitution path SHALL be configured in WORKFLOW.md so that it is automatically included in the agent's context window.

**User Story:** As a developer I want every AI action to respect my project's constitution automatically, so that generated code, specs, and documentation are consistent with established project conventions without me having to remind the AI each time.

#### Scenario: Skill reads constitution before execution
- **GIVEN** a WORKFLOW.md with `context` field referencing `openspec/CONSTITUTION.md`
- **WHEN** the developer invokes any skill command (e.g., `workflow propose`, `workflow apply`)
- **THEN** the agent loads and considers the constitution content before generating any output

#### Scenario: Artifact generation respects constitution
- **GIVEN** a constitution specifying "Commits: Imperative present tense with category prefix"
- **WHEN** the agent generates a tasks.md or design.md artifact
- **THEN** any commit messages or naming conventions referenced in the artifact follow the constitution's rules

#### Scenario: Constitution missing triggers warning
- **GIVEN** a project where `openspec/CONSTITUTION.md` does not exist
- **AND** WORKFLOW.md references it as a required context file
- **WHEN** a skill is invoked
- **THEN** the agent warns the user that the constitution is missing and recommends running `workflow init`

### Requirement: Constitution Update
The constitution SHALL be updated when the design phase introduces new technologies, patterns, or architectural changes that establish new project-wide conventions. During the design artifact generation, if the agent determines that the proposed design introduces a technology not listed in the Tech Stack, a new architectural pattern, or a convention that should apply globally, the agent SHALL update `openspec/CONSTITUTION.md` to reflect the addition. Updates SHALL be additive by default -- existing entries SHALL NOT be removed without explicit user approval. When a design proposes replacing an existing technology, the agent SHALL ask the user directly whether to replace the entry, rather than leaving a `<!-- REVIEW -->` marker. The agent SHALL note constitution changes in the design document so they are visible during review.

**User Story:** As a developer I want the constitution to evolve as my project grows, with replacements confirmed through direct questions rather than invisible markers.

#### Scenario: New dependency added to constitution during design
- **GIVEN** a design document that introduces Redis as a caching layer
- **AND** the current constitution does not mention Redis
- **WHEN** the agent generates or updates the design artifact
- **THEN** the agent adds Redis to the Tech Stack section of `openspec/CONSTITUTION.md` and notes this addition in the design document

#### Scenario: New architectural pattern captured
- **GIVEN** a design that introduces an event-driven messaging pattern between services
- **AND** the constitution's Architecture Rules do not mention event-driven patterns
- **WHEN** the agent processes the design
- **THEN** the agent adds the event-driven pattern to Architecture Rules with a description of where and how it applies

#### Scenario: Technology replacement resolved through direct question
- **GIVEN** a design that replaces Jest with Vitest for testing
- **WHEN** the agent updates the constitution
- **THEN** the agent asks the user: "The design replaces Jest with Vitest. Should I remove the Jest entry and add Vitest, or keep both?"
- **AND** the agent applies the user's decision without leaving a REVIEW marker

#### Scenario: Constitution changes documented in design
- **GIVEN** a design phase that triggers constitution updates
- **WHEN** the agent modifies `openspec/CONSTITUTION.md`
- **THEN** the design document includes a section or note listing the specific constitution changes made

### Requirement: Constitution Contains Only Project-Specific Rules
The constitution SHALL contain only rules that are specific to the project and not already defined by the schema or its templates. Rules that duplicate schema instructions, templates, or artifact `requires` chains SHALL be removed. The constitution retains: Tech Stack, Architecture Rules (structure and paths), Code Style (project conventions not in schema), Constraints (principles not enforced by schema mechanics), Conventions, and Standard Tasks (project-specific post-implementation steps appended to the universal standard tasks in the schema template). Standard Tasks that update the PR body SHALL include GitHub issue-closing keywords (e.g., `Closes #X`) when the change originated from a GitHub issue, to ensure issues are auto-closed on merge regardless of merge strategy.

**User Story:** As a workflow maintainer I want the constitution free of redundancy, so that each rule lives in exactly one authoritative place and the constitution stays focused on project-specific knowledge.

#### Scenario: Constitution does not duplicate schema-defined rules

- **GIVEN** the project constitution at `openspec/CONSTITUTION.md`
- **WHEN** its contents are compared with WORKFLOW.md and the Smart Templates in `openspec/templates/`
- **THEN** the constitution SHALL NOT contain rules about spec format, task format, assumption markers, capability naming, or artifact pipeline ordering
- **AND** those rules SHALL exist only in the Smart Templates' `instruction` fields or WORKFLOW.md

#### Scenario: Constitution retains project-specific content

- **GIVEN** the project constitution
- **WHEN** its sections are inspected
- **THEN** it SHALL contain Tech Stack, Architecture Rules, Code Style, Constraints, Conventions, and Standard Tasks
- **AND** Architecture Rules SHALL include directory paths and structural decisions
- **AND** Conventions SHALL include commit style, version bump, README accuracy, and friction tracking
- **AND** Standard Tasks SHALL contain project-specific post-implementation steps (or be empty with an explanatory comment)

#### Scenario: PR body update includes issue-closing keywords

- **GIVEN** a change that originated from GitHub issue #42
- **AND** the constitution defines a pre-merge standard task to update the PR body
- **WHEN** the agent executes the PR body update task
- **THEN** the updated PR body SHALL include `Closes #42`
- **AND** the issue SHALL be auto-closed when the PR is merged (regardless of merge strategy)

#### Scenario: No issue linked — keywords omitted

- **GIVEN** a change that did not originate from a GitHub issue
- **WHEN** the agent executes the PR body update task
- **THEN** the updated PR body SHALL NOT include issue-closing keywords

### Requirement: Friction Tracking Convention

The constitution's Conventions section SHALL include a rule requiring that workflow friction discovered during any workflow run be captured as a GitHub Issue with the `friction` label. The convention SHALL specify that each issue must include: what happened, expected behavior, and a suggested fix.

**User Story:** As a plugin maintainer I want a convention for capturing workflow friction systematically, so that problems discovered during dogfooding are tracked and addressed instead of being lost in conversation history.

#### Scenario: Friction convention present in constitution

- **GIVEN** the project constitution at `openspec/CONSTITUTION.md`
- **WHEN** the Conventions section is inspected
- **THEN** it SHALL contain a "Workflow friction" entry
- **AND** the entry SHALL require capturing friction as GitHub Issues with the `friction` label
- **AND** the entry SHALL specify including: what happened, expected behavior, and suggested fix

## Edge Cases

- **Friction in non-plugin projects**: The convention applies to all projects using the opsx workflow, not just the plugin itself. Consumer projects should track friction in their own issue trackers.
- **Multiple issues referenced**: If a change addresses multiple issues, all should be referenced (e.g., `Closes #42, Closes #43`).
- **New schema version adds rules**: When the schema is updated with new instructions, the constitution should be audited for newly-redundant entries.
- **Bootstrap on empty project**: A project with no source files or configuration. The agent SHALL generate a minimal constitution with placeholder sections and mark all entries with `<!-- REVIEW -->`.
- **Contradictory patterns observed**: The codebase contains two conflicting conventions (e.g., both camelCase and snake_case function names). The agent SHALL document both observed patterns and mark the entry with `<!-- REVIEW -->` for user resolution.
- **Constitution manually edited**: A user manually edits the constitution outside of the workflow. The agent SHALL treat the manual edits as authoritative and not overwrite them during subsequent bootstrap or design updates.
- **Multiple design phases updating constitution**: Sequential changes each add to the constitution. Later updates SHALL preserve additions from earlier changes and not regress them.
- **Large monorepo with mixed tech stacks**: A monorepo containing both Python and TypeScript projects. The agent SHALL document all observed tech stacks and note which directories each applies to.
- **Template file missing**: If the schema's `templates/constitution.md` does not exist, the bootstrap skill SHALL fall back to a hardcoded default section list and log a warning. This ensures backward compatibility with schemas that have not yet adopted the template.
- **Empty template**: If the template file exists but is empty, the bootstrap skill SHALL treat it as if no sections are defined and warn the user.
- **Custom sections in template**: If a future schema adds non-standard sections to its constitution template, the bootstrap skill SHALL generate content for all sections present in the template.

## Assumptions

- The constitution file is always located at `openspec/CONSTITUTION.md` relative to the project root. <!-- ASSUMPTION: Fixed path convention established by the plugin -->
- The WORKFLOW.md `context` field is configured during `workflow init` to reference the constitution. <!-- ASSUMPTION: Setup skill configures WORKFLOW.md correctly -->
- Constitution updates during design are immediately visible to subsequent skill invocations within the same session. <!-- ASSUMPTION: File system writes are synchronous and the agent re-reads context files -->
- The active schema is resolved before the bootstrap skill reads the constitution template. <!-- ASSUMPTION: Schema resolution happens in bootstrap Step 1 -->
- The template file path follows the existing convention: `openspec/templates/constitution.md`. <!-- ASSUMPTION: Consistent template directory structure -->
