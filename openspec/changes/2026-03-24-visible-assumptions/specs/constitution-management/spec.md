## MODIFIED Requirements

### Requirement: Bootstrap-Generated Constitution
The project constitution (`openspec/constitution.md`) SHALL be generated by the `/opsx:bootstrap` command based on observed patterns, conventions, and technologies found in the existing codebase. The agent SHALL scan source files, configuration files, directory structures, and dependency manifests to infer the tech stack, architecture rules, code style, constraints, and conventions. The constitution SHALL NOT contain invented or aspirational rules -- every entry MUST be traceable to an observed pattern in the codebase. Items where the agent is uncertain about the convention or its scope SHALL be marked with a `<!-- REVIEW -->` comment. After generation, the agent SHALL iterate through all REVIEW markers, present each to the user for confirmation, document the user's decision, and remove the marker. No REVIEW markers SHALL remain in the final constitution after bootstrap completes.

**User Story:** As a developer bootstrapping an existing project I want the constitution to reflect what my codebase actually does, with all uncertainties resolved through direct questions rather than invisible markers left for me to discover.

#### Scenario: Constitution generated from codebase scan
- **GIVEN** an existing project with TypeScript source files, a `tsconfig.json`, ESLint configuration, and a `package.json`
- **WHEN** the developer runs `/opsx:bootstrap`
- **THEN** the generated `openspec/constitution.md` includes TypeScript under Tech Stack, references the ESLint rules under Code Style, and captures the project's module structure under Architecture Rules

#### Scenario: Uncertain conventions resolved through user interaction
- **GIVEN** a codebase where some files use tabs and others use spaces for indentation
- **WHEN** the agent generates the constitution
- **THEN** the agent initially marks the indentation entry with `<!-- REVIEW -->`
- **AND** the agent presents the inconsistency to the user: "Indentation convention unclear — both tabs and spaces observed. Which should be the standard?"
- **AND** after the user responds, the agent updates the entry with the user's decision and removes the `<!-- REVIEW -->` marker

#### Scenario: No invented rules
- **GIVEN** a Python project with no test framework configured
- **WHEN** the agent generates the constitution
- **THEN** the constitution does NOT include a testing convention or framework entry, because none was observed in the codebase

#### Scenario: Constitution covers standard sections
- **GIVEN** a codebase with identifiable patterns
- **WHEN** the agent generates the constitution
- **THEN** the resulting constitution contains at minimum: Tech Stack, Architecture Rules, Code Style, Constraints, and Conventions sections

#### Scenario: No REVIEW markers remain after bootstrap
- **GIVEN** a completed bootstrap run
- **WHEN** the final constitution is inspected
- **THEN** zero `<!-- REVIEW -->` markers remain in the file
- **AND** every uncertain item has been resolved with the user's explicit decision

### Requirement: Constitution Update
The constitution SHALL be updated when the design phase introduces new technologies, patterns, or architectural changes that establish new project-wide conventions. During the design artifact generation, if the agent determines that the proposed design introduces a technology not listed in the Tech Stack, a new architectural pattern, or a convention that should apply globally, the agent SHALL update `openspec/constitution.md` to reflect the addition. Updates SHALL be additive by default -- existing entries SHALL NOT be removed without explicit user approval. When a design proposes replacing an existing technology, the agent SHALL ask the user directly whether to replace the entry, rather than leaving a `<!-- REVIEW -->` marker. The agent SHALL note constitution changes in the design document so they are visible during review.

**User Story:** As a developer I want the constitution to evolve as my project grows, with replacements confirmed through direct questions rather than invisible markers.

#### Scenario: New dependency added to constitution during design
- **GIVEN** a design document that introduces Redis as a caching layer
- **AND** the current constitution does not mention Redis
- **WHEN** the agent generates or updates the design artifact
- **THEN** the agent adds Redis to the Tech Stack section of `openspec/constitution.md` and notes this addition in the design document

#### Scenario: New architectural pattern captured
- **GIVEN** a design that introduces an event-driven messaging pattern between services
- **AND** the constitution's Architecture Rules do not mention event-driven patterns
- **WHEN** the agent processes the design
- **THEN** the agent adds the event-driven pattern to Architecture Rules with a description of where and how it applies

#### Scenario: Technology replacement resolved through direct question
- **GIVEN** a design that replaces Jest with Vitest for testing
- **WHEN** the agent updates the constitution
- **THEN** the agent asks the user: "The design replaces Jest with Vitest. Should I remove the Jest entry and add Vitest, or keep both?"
- **AND** the agent applies the user's decision without leaving a REVIEW marker

#### Scenario: Constitution changes documented in design
- **GIVEN** a design phase that triggers constitution updates
- **WHEN** the agent modifies `openspec/constitution.md`
- **THEN** the design document includes a section or note listing the specific constitution changes made
