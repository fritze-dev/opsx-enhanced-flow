# Technical Design: Symphony WORKFLOW.md Repository Contract

## Context

opsx-enhanced-flow currently splits project configuration across two files: `openspec/constitution.md` (free-form prose rules read by the AI) and `openspec/config.yaml` (minimal YAML with schema reference). This split has two weaknesses: settings are buried in prose (not machine-readable), and there is no prompt template mechanism to render context-specific guidance.

OpenAI Symphony's `WORKFLOW.md` concept demonstrates a clean solution: a single, repository-owned file with YAML front matter for structured settings and a markdown body as an agent prompt template. This design document describes how to bring this pattern to opsx-enhanced-flow.

**Stakeholders**: Plugin maintainers (Fission-AI), project teams using the plugin, AI agents (Claude) consuming the workflow contract.

**Constraints**:
- Skills in `skills/` are generic plugin code and MUST NOT be modified for project-specific behavior (per constitution)
- No new external runtime dependencies (YAML parsing is native to the agent; template substitution is string-based)
- Must remain backward-compatible with existing `constitution.md` + `config.yaml` setups

---

## Architecture & Components

### File: `WORKFLOW.md` (project root)
New project-level file with two sections:
1. **YAML front matter** (optional, delimited by `---`) — structured settings
2. **Markdown body** — agent behavioral context and prompt template

```
[project root]/
  WORKFLOW.md          ← new authoritative contract (replaces constitution.md + config.yaml)
  openspec/
    config.yaml        ← legacy, retained for backward compat (ignored if WORKFLOW.md present)
    constitution.md    ← legacy, retained for backward compat (ignored if WORKFLOW.md present)
    schemas/opsx-enhanced/schema.yaml   ← no change
    specs/             ← no change
    changes/           ← no change
```

### Component: Agent Context Loader (behavioral, not code)
The AI agent already reads `openspec/constitution.md` and `openspec/config.yaml` at the start of every skill invocation (per the `context` field in `config.yaml`). The new behavior:

1. Check for `WORKFLOW.md` at the project root
2. If found: parse YAML front matter → apply settings; render markdown body with template variables → use as behavioral context
3. If not found: fall back to existing `constitution.md` + `config.yaml` behavior

This is a behavioral change to how skills start, not a code change. It is implemented via the `openspec/config.yaml` `context` field, which instructs the agent to read `WORKFLOW.md` (when present) as the primary context source.

### Component: `/opsx:init` Skill
The init skill creates project scaffolding. Currently generates `openspec/config.yaml` + `openspec/constitution.md`. Modified behavior:
- Generate `WORKFLOW.md` with template content (front matter + markdown body)
- Detect legacy setups and provide migration guidance
- Preserve existing `WORKFLOW.md` if already present (idempotent)

### Component: `openspec/config.yaml` (`context` field)
The `context` field in `config.yaml` is what tells the AI agent where to look for project rules. Updated to instruct the agent to prefer `WORKFLOW.md` over `constitution.md`:

```yaml
schema: opsx-enhanced
context: |
  If WORKFLOW.md exists at the project root, read it as the primary project workflow contract
  (parse YAML front matter for settings, render markdown body as behavioral context).
  If WORKFLOW.md is absent, fall back to openspec/constitution.md.
```

### Template Variable Substitution
Minimal string substitution applied to the WORKFLOW.md markdown body before use:
- `{{ change.name }}` → current change directory name (from `openspec/changes/<name>/`)
- `{{ change.stage }}` → current pipeline artifact stage (passed by the invoking skill context)
- `{{ project.name }}` → value of `project.name` from YAML front matter

Implementation: simple `str.replace("{{ variable }}", value)` substitution, not a full template engine. Unknown tokens left as-is.

## Goals & Success Metrics

| Metric | Target | Verification |
|--------|--------|-------------|
| WORKFLOW.md generated by `/opsx:init` | File exists at project root after init | PASS: `test -f WORKFLOW.md` after fresh init |
| WORKFLOW.md supersedes constitution.md | Agent uses WORKFLOW.md body when both present | PASS: Agent references WORKFLOW.md in response context |
| Backward compat: legacy setup unchanged | Projects without WORKFLOW.md work identically to before | PASS: Existing project behavior unchanged in regression test |
| YAML front matter parsed correctly | `schema`, `project.name`, `docs_language` fields applied | PASS: Agent acknowledges project name from front matter |
| Template variable `{{ change.name }}` resolves | Variable substituted at skill invocation | PASS: AI context shows resolved variable, not raw token |
| Migration path documented | `/opsx:init` shows migration guidance for legacy projects | PASS: Init output includes migration instructions for legacy projects |

---

## Non-Goals

- Full Liquid template engine — minimal `{{ }}` substitution only
- Real-time file watching / dynamic config reload — not applicable to a Claude Code plugin  
- Changes to the OpenSpec CLI (`@fission-ai/openspec`) — WORKFLOW.md is an agent convention
- Automatic deletion of `constitution.md` or `config.yaml` during migration — retained for safety

---

## Decisions

| # | Decision | Rationale | Alternatives Considered |
|---|----------|-----------|------------------------|
| 1 | `WORKFLOW.md` at project root (not `openspec/WORKFLOW.md`) | Consistent with Symphony; root-level placement signals it's a project-level, not tool-level, contract | `openspec/WORKFLOW.md` (too tool-specific), `AGENT.md` (less descriptive) |
| 2 | YAML front matter rather than a separate YAML file | Single file, less cognitive overhead; standard markdown convention (Jekyll, Hugo, etc.) | Separate `openspec/workflow-config.yaml` file |
| 3 | Backward compatibility via fallback, not versioning | Non-breaking; existing projects don't need immediate migration | Require migration script; version flag in WORKFLOW.md |
| 4 | Config.yaml `context` field drives agent behavior | Skills remain generic (per constitution constraint); behavior change happens at the config/context level without touching skill files | Skill-level conditional (would violate "no skill modifications for project-specific behavior") |
| 5 | Minimal `{{ }}` substitution over full Liquid | Zero new dependencies; sufficient for initial use cases; full Liquid can be added later if needed | Jinja2, Handlebars, Liquid library |

## Risks & Trade-offs

- **[Risk] Migration complexity for existing projects** → Mitigation: Keep `constitution.md` and `config.yaml` as fallback; provide clear migration guidance in `/opsx:init` output and spec documentation
- **[Risk] YAML front matter parsing errors silently degrade behavior** → Mitigation: Agent reports YAML parse warning and falls back to plain-body mode; never silently ignores errors
- **[Risk] Template variables used in existing WORKFLOW.md content that aren't in the supported list** → Mitigation: Unknown tokens left as-is (not substituted), so existing markdown content using `{{ }}` for other purposes is preserved
- **[Risk] `openspec/config.yaml` `context` field is long-form prose; updates may drift** → Mitigation: The updated context field instruction is short and declarative; document the expected format in the constitution

---

## Migration Plan

For new projects:
1. Run `/opsx:init` → generates `WORKFLOW.md` automatically

For existing projects (migration):
1. Run `/opsx:init` → detects legacy setup, generates `WORKFLOW.md` from existing `constitution.md` content
2. Review generated `WORKFLOW.md` — merge any custom rules from `constitution.md`
3. Update `openspec/config.yaml` `context` field to reference `WORKFLOW.md`
4. (Optional) Remove or archive `constitution.md` and legacy `config.yaml` entries once satisfied

**Rollback**: Delete `WORKFLOW.md`. System falls back to `constitution.md` automatically.

## Open Questions

1. Should `/opsx:init` automatically populate `project.name` in the WORKFLOW.md front matter by detecting the project name (e.g., from `package.json`, git remote, or directory name)? → Recommended: yes, with a fallback to a placeholder.
2. Should the `openspec/config.yaml` `context` field be updated as part of this change, or should the schema template be updated instead? → The schema template in `openspec/schemas/opsx-enhanced/` should be updated so new inits get the correct context by default.
