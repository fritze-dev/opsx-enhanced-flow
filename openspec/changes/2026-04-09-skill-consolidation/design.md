# Technical Design: Skill Consolidation

## Context

The opsx-enhanced plugin currently has 11 SKILL.md files (1,593 lines) that duplicate 60-70% of spec content. Shared logic like change context detection is copy-pasted across 6 skills. PR #97 validated the concept of action templates with sub-agents but created separate action template files. This change consolidates to 4 commands with inline actions in WORKFLOW.md and a single router skill.

Affected systems:
- `src/skills/` — 11 SKILL.md files to be replaced by 1 router
- `openspec/WORKFLOW.md` — gains `actions:` section, pipeline expanded to 7 stages
- `openspec/templates/` — gains `review.md` template
- `openspec/specs/` — 8 specs modified, 1 removed
- `.claude-plugin/marketplace.json` + `src/.claude-plugin/plugin.json` — command registration updated
- `openspec/CONSTITUTION.md` — three-layer vocabulary updated

## Architecture & Components

### Router (`src/skills/opsx/SKILL.md`)

Single SKILL.md file (~80 lines) that handles all 4 commands:

```
User invokes /opsx:<command>
  → Router: intent recognition
  → Router: change context detection (shared)
  → Router: WORKFLOW.md loading (shared)
  → Dispatch:
      propose → pipeline traversal (artifact generation loop)
      apply   → sub-agent with actions.apply instruction + specs
      finalize → sub-agent with actions.finalize instruction + specs
      init    → sub-agent with actions.init instruction + specs (no change context)
```

The router reads `actions.<name>` from WORKFLOW.md frontmatter, loads specs listed in `specs:` array, and spawns a sub-agent with bounded context (instruction + specs + change artifacts). The sub-agent does NOT receive the router's conversation history.

### WORKFLOW.md Structure

```yaml
---
template-version: 3
templates_dir: openspec/templates
pipeline: [research, proposal, specs, design, preflight, tasks, review]

actions:
  init:
    specs: [project-setup, project-bootstrap, constitution-management, quality-gates]
    instruction: |
      ...
  apply:
    specs: [task-implementation, quality-gates]
    instruction: |
      ...
  finalize:
    specs: [release-workflow, user-docs, architecture-docs, decision-docs]
    instruction: |
      ...

worktree: { enabled: true, path_pattern: .claude/worktrees/{change}, auto_cleanup: true }
automation:
  post_approval:
    action: finalize
    labels: { running: automation/running, complete: automation/complete, failed: automation/failed }
---
# Workflow
...body sections (Context, Post-Artifact Hook)...
```

### review.md Template

New artifact template at `openspec/templates/review.md`:
- `generates: review.md`
- `requires: [tasks]`
- Template body defines report structure: Summary scorecard, Findings (CRITICAL/WARNING/SUGGESTION), Verdict (PASS/PASS WITH WARNINGS/FAIL)
- Generated by apply as QA-loop output, not during artifact-forward generation
- Replaces transient verify report with persistent, PR-visible artifact

### Plugin Registration

Claude Code plugin system registers skills via SKILL.md frontmatter. Options for 4 commands from 1 router:

**Approach: 4 stub files + 1 shared router body**

```
src/skills/
├── propose/SKILL.md    ← frontmatter: name: propose, then "Read and follow src/skills/router.md"
├── apply/SKILL.md      ← frontmatter: name: apply, then "Read and follow src/skills/router.md"
├── finalize/SKILL.md   ← frontmatter: name: finalize, then "Read and follow src/skills/router.md"
├── init/SKILL.md       ← frontmatter: name: init, then "Read and follow src/skills/router.md"
└── router.md           ← shared router logic (~80 lines, no frontmatter)
```

Each stub is ~5 lines (frontmatter + include directive). The router logic lives once in `router.md`. This works within the plugin system's 1-command-per-SKILL.md constraint.

## Goals & Success Metrics

- Commands reduced from 13 to 4: PASS if only init, propose, apply, finalize are registered
- Orchestration code reduced by >90%: PASS if router.md + stubs + WORKFLOW.md actions < 200 lines total
- review.md generated during apply: PASS if apply produces review.md in the change directory
- All existing spec behavior preserved: PASS if apply + propose produce equivalent results to current apply + ff + verify
- Pipeline includes review: PASS if `pipeline` array in WORKFLOW.md has 7 entries ending with `review`

## Non-Goals

- Spec merges (e.g., merging project-setup + project-bootstrap into one spec)
- Consumer migration tooling (manual `/opsx:init` sufficient)
- Closing PR #97 (done after this change lands)

## Decisions

| Decision | Rationale | Alternatives |
|----------|-----------|--------------|
| Actions inline in WORKFLOW.md, not separate template files | Actions have no body (no output structure) — separate files add maintenance without benefit. PR #97 showed separate files work but are unnecessary overhead | Separate action template files (PR #97 approach); action definitions in specs |
| 4 stub SKILL.md files + shared router.md | Plugin system requires 1 SKILL.md per command. Stubs are minimal (5 lines each), router logic shared. Total: ~100 lines vs 1,593 | Single SKILL.md with all commands (plugin system may not support); all logic in stubs (duplication) |
| review.md as pipeline artifact | Not skippable (file existence check), PR-visible (in change directory), eliminates QA-loop checkbox hacking. research.md opens the change, review.md closes it | Transient verify report (current — lost after conversation); verify.md naming (less intuitive in PR context) |
| Eliminate discover entirely | 0/49 changes used it. Research is the first pipeline artifact generated by propose. Q&A happens naturally in conversation | Keep as --discover flag; keep as separate command |
| template-version 3 (breaking) | Major architectural change warrants version bump. Consumers need `/opsx:init` to migrate | template-version 2 (incremental, but PR #97 already claimed v2) |

## Risks & Trade-offs

- **Thin instruction may be insufficient for complex actions** → Mitigation: test apply (most complex) first. If 10-line instruction + spec reference produces unreliable results, expand instruction or add action-specific context loading in the router.
- **Plugin stub indirection** → Mitigation: stubs are trivially simple ("Read and follow router.md"). If plugin system doesn't support cross-file includes, inline the router into each stub (increases from ~100 to ~320 lines, still 80% reduction).
- **review.md staleness after manual code changes** → Mitigation: apply deletes review.md at start, regenerates at end. Propose can re-run individual steps.
- **Breaking change for consumers on template-version 1** → Mitigation: `/opsx:init` handles migration. template-version 3 is an explicit signal.

### GitHub Actions Pipeline (`.github/workflows/pipeline.yml`)

CI workflow triggered on PR approval that runs finalize automatically:
- Triggered by `pull_request_review` event with `state == 'approved'`
- Checks all required reviews are approved
- Uses `claude-code-action` to read WORKFLOW.md `automation.post_approval` config
- Executes the finalize action (changelog → docs → version-bump)
- Manages labels (`automation/running`, `automation/complete`, `automation/failed`)
- Commits and pushes results to PR branch

The action reads `automation.post_approval.action: finalize` from WORKFLOW.md — the action name maps to `actions.finalize` which defines specs and instruction. This means CI behavior is driven by the same WORKFLOW.md config as local execution.
