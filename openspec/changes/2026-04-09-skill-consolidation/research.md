### Usage Data: discover

Checked all 49 changes in `openspec/changes/`:
- 49/49 have `research.md` generated by `/opsx:ff`, never by `/opsx:discover`
- Only 1/49 `research.md` files has actual Decision entries recorded
- The Q&A feature (Impact × Uncertainty scoring, max 5 questions) was never adopted
- Conclusion: discover can be eliminated entirely

### PR #97 (pr-lifecycle-automation)

PR #97 on branch `pr-lifecycle-automation` already partially implements this direction:
- Created Action Templates (`type: action`) for apply, verify, changelog, docs, version-bump
- Extended pipeline to include action steps
- ff spawns sub-agents for action steps with template instruction + bounded context
- Added CI automation config (`automation.post_approval`)

Key insight from PR #97: Action Templates work as self-contained instructions for sub-agents. This validates the approach of specs + thin instruction replacing SKILL.md files.

PR #97 will be closed after this change lands — its concepts are incorporated here with a cleaner architecture (actions inline in WORKFLOW.md, no separate action template files).

## 2. External Research

### Fission-AI OpenSpec (github.com/Fission-AI/OpenSpec)

The upstream OpenSpec project uses similar patterns:
- `propose` as the main entry command (validates our rename from ff)
- `explore` for unclear requirements (equivalent to our discover — also rarely used in practice)
- Core profile with only 3 commands: `propose → apply → archive`
- Philosophy: "Actions, not phases" — commands are things you do, not stages you're stuck in
- Two modes: core (simple) vs expanded (full control)

Key alignment: our 4-command target (init, propose, apply, finalize) closely mirrors Fission's core profile approach.

## 3. Approaches

| Approach | Pro | Contra |
|----------|-----|--------|
| **A: Router + inline actions in WORKFLOW.md** | 93% reduction; single source of truth; reuses existing template pattern; actions defined where pipeline is defined | instruction fields might be too thin for complex actions; WORKFLOW.md grows ~70 lines |
| B: Router + separate action template files | Cleaner separation of concerns | Creates new files that need maintenance; PR #97 showed this adds complexity without benefit |
| C: Keep skills, deduplicate shared logic | Lower risk, incremental | Doesn't solve fundamental duplication; still 11 files to maintain |
| D: Eliminate skills, rely purely on specs | Maximum simplicity | Specs are declarative, not procedural — Claude needs procedural guidance for reliable execution |

**Recommended: Approach A** — Router + inline actions. Validated by PR #97's sub-agent pattern. Actions in WORKFLOW.md reuse the same `instruction` pattern as Smart Templates.

## 4. Risks & Constraints

- **Claude reliability with thin instructions**: verify currently has 223 lines of procedural guidance. With ~10 lines instruction + spec reference, Claude must derive procedure from declarative spec requirements. Mitigation: test apply (most complex) first.
- **Plugin system registration**: Claude Code registers 1 command per SKILL.md. A single router needs to register all 4 commands. May need stub files.
- **review.md staleness**: If code changes after review.md is generated, it becomes stale. Mitigation: apply deletes review.md at start, regenerates at end.
- **Constitution references "Three-layer architecture"**: The constitution and several specs reference the current three-layer model (Constitution → WORKFLOW.md + Templates → Skills). This vocabulary needs updating — skills layer becomes "Router + Actions."
- **Backward compatibility**: Consumers using current WORKFLOW.md template-version 1 need migration path. `template-version: 3` signals the breaking change.
