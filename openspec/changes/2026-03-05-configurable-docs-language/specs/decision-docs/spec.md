## ADDED Requirements

### Requirement: Language-Aware ADR Generation
ADR files generated by `/opsx:docs` SHALL respect the `docs_language` setting from `openspec/config.yaml`. When a non-English language is configured, all ADR section headings (Status, Context, Decision, Rationale, Alternatives Considered, Consequences, References) and descriptive content SHALL be translated to the target language. ADR file names (`adr-NNN-<slug>.md`) SHALL remain in English. The slug SHALL always be derived from the English Decision column text regardless of output language.

**User Story:** As a non-English-speaking team I want ADRs in my language, so that architectural decisions are understandable without translation.

#### Scenario: ADR generated in configured language
- **GIVEN** `openspec/config.yaml` contains `docs_language: German`
- **AND** an archived design.md with a Decisions table exists
- **WHEN** the developer runs `/opsx:docs`
- **THEN** ADR headings SHALL be in German (e.g., "Kontext", "Entscheidung", "Begründung", "Konsequenzen")
- **AND** ADR content SHALL be in German
- **AND** the file name SHALL remain in English (e.g., `adr-001-use-config-yaml.md`)

#### Scenario: ADR consequences translated
- **GIVEN** `docs_language: Spanish`
- **WHEN** the agent generates ADR Consequences
- **THEN** subsection headings SHALL be "Positivas" and "Negativas" instead of "Positive" and "Negative"

## Edge Cases

- **Slug derivation**: The slug is always derived from the original English Decision column text in design.md, never from translated content. This ensures stable file names across language changes.

## Assumptions

No assumptions made.
