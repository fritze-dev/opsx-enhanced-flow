# Technical Design: Configurable Documentation Language

## Context

All user-facing documentation generated by `/opsx:docs` and `/opsx:changelog` is currently English-only. The plugin serves non-English-speaking teams who need documentation in their own language. The change adds a `docs_language` configuration field and teaches two skills to use it, while ensuring internal workflow artifacts remain English.

Three skill files are affected. No schema, template, or other skill changes required.

## Architecture & Components

| Component | Change | File |
|-----------|--------|------|
| Init skill | Add `docs_language` (commented out) + English enforcement to config template | `skills/init/SKILL.md` |
| Docs skill | Add Step 0 (language determination) + translation reminders in Steps 3, 4, 5 | `skills/docs/SKILL.md` |
| Changelog skill | Add Step 0 (language determination) + translation reminder in Step 6 | `skills/changelog/SKILL.md` |

**Data flow:** `openspec/config.yaml` → skill reads `docs_language` at generation time → applies translation to output files.

## Goals & Success Metrics

* Init skill config template includes commented-out `docs_language` field and English enforcement in `context` — PASS/FAIL
* Docs skill has Step 0 that reads and applies `docs_language` — PASS/FAIL
* Changelog skill has Step 0 that reads and applies `docs_language` — PASS/FAIL
* Missing `docs_language` field defaults to English (backward-compatible) — PASS/FAIL

## Non-Goals

- Translating doc templates (`templates/docs/`) — remain English structural guides
- Adding language support to schema.yaml — language is a generation-time concern, not a pipeline concern
- Translating workflow artifacts (research, proposal, specs, design, preflight, tasks) — always English
- Per-skill language parameters — single config field covers all doc output
- Language validation or ISO codes — LLM handles free-text language names reliably
- Translating the constitution — it is an internal artifact

## Decisions

| # | Decision | Rationale | Alternatives Considered |
|---|----------|-----------|------------------------|
| 1 | Single `docs_language` field in config.yaml | Central, backward-compatible, read by all skills via existing config loading | Per-skill parameter (duplication), environment variable (not persistent), constitution entry (mixes concerns) |
| 2 | Commented-out field in init template for discoverability | Users discover the feature without it being active by default. Existing projects are unaffected. | Active field defaulting to English (unnecessary noise in config), separate config documentation (not discoverable) |
| 3 | English enforcement via config `context` field | Context is passed to all skills automatically — single enforcement point, no per-skill duplication | Per-skill instruction (requires updating every skill), schema-level rule (language is not a pipeline concern) |
| 4 | Translation at generation time, not in templates | Templates are structural guides. Translation during generation allows one set of templates for all languages. | Per-language template sets (template proliferation, maintenance burden) |

## Risks & Trade-offs

- **LLM translation quality varies by language** → Acceptable: major languages (German, French, Spanish, etc.) produce high-quality results. Exotic languages may need manual review.
- **No runtime validation of `docs_language` value** → Mitigated: LLMs handle fuzzy language names well. Worst case: unrecognizable value falls back to English with a warning.
- **Language change mid-project leaves mixed-language changelog** → Acceptable: existing entries are preserved (guardrail), new entries use new language. Documented as expected behavior.
