# Research: Context Loading Guardrails

## 1. Current State

The project has a growing inventory of readable artifacts with no central guidance on what Claude should or should not load during workflow actions:

| Source | ~Lines | Purpose | Current guidance |
|--------|--------|---------|-----------------|
| `openspec/constitution.md` | 54 | Global rules | Always loaded via config.yaml |
| `openspec/specs/*/spec.md` (18) | ~2,800 | Source of truth | Read ad-hoc by skills |
| `openspec/changes/archive/` (26 dirs) | ~13,000 | Historical record | None |
| `docs/README.md` | 148 | Architecture overview | None |
| `docs/capabilities/*.md` (18) | ~1,500 | User-facing docs | None |
| `docs/decisions/adr-*.md` (27) | ~1,300 | Design decisions | None |

**Problem**: Without guardrails, Claude may load archive (~13k lines), capability docs (redundant with specs), or all 27 ADRs when context window space would be better spent on actual work.

**Affected files**:
- `openspec/constitution.md` — needs new section
- `openspec/schemas/opsx-enhanced/schema.yaml` — research instruction needs ADR guidance

**Prior attempt**: PR #52 tried to solve the ADR-awareness part of this by splitting docs/README.md into 3 files and making `/opsx:discover` deep-dive into ADRs — 29 files changed, 1,209 additions. Rejected as too complex.

## 2. External Research

Not applicable — this is an internal project convention change.

## 3. Approaches

| Approach | Pro | Contra |
|----------|-----|--------|
| **A: Constitution section** — add `## Context Guardrails` to constitution with read/don't-read rules | Simple (~15 lines), auto-loaded before every action, consistent with existing convention-based enforcement | Convention-based, not hard-enforced |
| B: Per-stage schema rules — add `context:` blocks to each artifact in schema.yaml | Stage-specific precision | Verbose (6 artifacts × N rules), redundant across stages, over-engineered |
| C: Separate guardrails file — new `openspec/guardrails.md` | Clean separation | Another file to maintain, not auto-loaded by config.yaml |

**Recommendation**: Approach A. The guardrails are project-level knowledge ("archive is historical, capability docs are derived") not stage-specific. The constitution is already loaded before every action.

## 4. Risks & Constraints

- **Convention-based enforcement**: Claude may still read excluded sources. Consistent with all other enforcement in the project (ADR-004, ADR-006). The explicit "Do Not Read" language is strong guidance.
- **Constitution growth**: Adding ~15 lines to 54-line file (~28% increase). Still well under 100 lines.
- **ADR index staleness**: The Key Design Decisions table in docs/README.md is regenerated by `/opsx:docs`, so it stays current after archiving.

## 7. Decisions

| # | Decision | Rationale | Alternatives Considered |
|---|----------|-----------|------------------------|
| 1 | Add `## Context Guardrails` section to constitution | Auto-loaded before every action, consistent with convention-based enforcement pattern | Per-stage schema rules (too verbose), separate file (not auto-loaded) |
| 2 | Use three tiers: Always Read / When Relevant / Do Not Read | Clear, scannable, actionable categories | Binary read/don't-read (loses nuance for ADRs), per-skill rules (too granular) |
| 3 | Enrich schema research instruction with ADR-via-README guidance | Directly addresses Issue #17 without modifying skills | Modify discover SKILL.md (violates skill immutability), read all ADRs (wastes context) |
| 4 | Place section after Architecture Rules, before Code Style | Guardrails describe how to navigate the architecture — logically follows Architecture Rules | After Conventions (too late), before Tech Stack (illogical) |
