## Why

Assumption markers in specs use two patterns inconsistently: Pattern A (visible list item + hidden tag) and Pattern B (everything hidden in HTML comment). Pattern B makes assumptions invisible in Markdown preview, breaking review and preflight audits. Additionally, `<!-- REVIEW -->` markers placed by bootstrap and docs skills are never actively resolved — they sit invisibly in files until someone reads the raw source.

## What Changes

- **ASSUMPTION format**: Enforce Pattern A (`- Visible text. <!-- ASSUMPTION: tag -->`) as the standard. Convert all Pattern B occurrences (`<!-- ASSUMPTION: ... -->` with no visible text) to Pattern A via delta specs for all affected baseline specs.
- **REVIEW auto-resolution**: Add active handling to skills that write `<!-- REVIEW -->` markers — the system must iterate through markers, ask the user, document the decision, and delete the marker. No format change needed (stays as HTML comment).
- **Preflight expansion**: Extend Section F from "Assumption Audit" to "Marker Audit" — audit ASSUMPTION markers (rated as Acceptable/Clarification/Blocking) and flag any remaining `<!-- REVIEW -->` markers as Blocking.
- **Schema/Templates**: Document and enforce the correct assumption format in schema instructions and templates.

## Capabilities

### New Capabilities

None.

### Modified Capabilities

- `spec-format`: Add requirement enforcing Pattern A for assumptions. The correct format is `- Visible assumption text. <!-- ASSUMPTION: short tag -->`. Pattern B (everything inside HTML comment) is a format violation.
- `quality-gates`: Expand preflight Section F to "Marker Audit" — audit both ASSUMPTION markers (rated) and REVIEW markers (always Blocking if present). Search pattern stays `<!-- ASSUMPTION` for tags, but verify visible text accompanies each tag.
- `constitution-management`: Bootstrap skill must actively resolve `<!-- REVIEW -->` markers — iterate through each, ask user for confirmation, document the decision, and delete the marker. Scenarios updated to reflect auto-resolution behavior.
- `decision-docs`: Docs skill must actively resolve `<!-- REVIEW: ... -->` markers during ADR reference validation — present broken links to user, resolve, and remove markers instead of leaving them in the file.

- Archived changes (immutable — old format stays)
