## Why

Plugin consumers can't detect updates because `/plugin update` requires a version bump in `plugin.json` — currently done manually and regularly forgotten. Additionally, `marketplace.json` version is out of sync (1.0.0 vs 1.0.3). There's no documented release process and no end-to-end test for the install/update flow.

## What Changes

- **Post-archive auto-bump convention:** Constitution gains a new convention that instructs `/opsx:archive` to auto-increment the patch version in both `plugin.json` and `marketplace.json` after archiving
- **Version sync fix:** `marketplace.json` version synced to match `plugin.json`
- **Release documentation:** Specs cover all release processes (auto patch, manual minor/major, consumer update) — `/opsx:docs` generates user-facing docs automatically
- **Constitution update:** Manual version bump convention replaced with auto-bump convention + skill immutability rule
- **README update:** Improved "Updating the Plugin" section with clearer consumer instructions

## Capabilities

### New Capabilities
- `release-workflow`: Defines the version management strategy — auto patch-bump convention, manual minor/major process, consumer update instructions, and E2E test checklist

### Modified Capabilities
_None — no spec-level behavior changes. The archive behavior is extended via a constitution convention, not via spec modification._

## Impact

- `openspec/constitution.md` — new post-archive convention + skill immutability rule
- `.claude-plugin/plugin.json` — version auto-incremented on each archive (via convention)
- `.claude-plugin/marketplace.json` — version synced on each archive + immediate fix
- `README.md` — update instructions improved
- `docs/capabilities/release-workflow.md` — auto-generated by `/opsx:docs`

## Scope & Boundaries

**In scope:**
- Post-archive auto-bump convention in constitution
- Skill immutability rule in constitution (Issue #10)
- marketplace.json version sync (immediate fix + ongoing via convention)
- Release workflow fully spec'd — docs auto-generated via `/opsx:docs`
- Brief README note about release workflow

**Out of scope:**
- Skill file modifications (skills are shared, never project-specific)
- `/opsx:release` skill (deferred — auto-bump covers 95% of cases)
- `/opsx:status` skill (separate feature)
- Git hooks or CI/CD automation
- Automatic git tagging (manual for minor/major releases)
- Review checkpoint convention in ff-workflow (Issue #9, separate change)
